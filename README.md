# Mindustry Web Port

Mindustry `v157.4` running in the browser through CheerpJ plus a custom JS SDL/WebGL bridge.

Notes:
- `webpage_content_reporter.js` "Cannot use import statement outside a module" is browser-extension noise, not repo code.
- Mindustry's release JARs are Java 8 bytecode (class file version 52), so we let CheerpJ run them on its default Java 8 runtime. Pinning Java 17 made CheerpJ fetch `/lib/modules` (the JDK 9+ jimage), which fails with `ERR_CACHE_OPERATION_NOT_SUPPORTED` and surfaces as a misleading `NoClassDefFoundError: netscape/javascript/JSObject`.
- Mindustry's `DesktopLauncher.checkJavaVersion()` fatally errors when `arc.util.OS.javaVersionNumber < 17`. CheerpJ ignores all of our `java.version` overrides (`javaProperties`, runtime `System.setProperty`) and the override classpath approach doesn't work (Vercel can't serve directory listings, so CheerpJ's probe of `/override` 404s and the entry is dropped). `build.sh` patches both JARs in place: `Mindustry.jar`'s `DesktopLauncher.checkJavaVersion` has its first opcode flipped from `getstatic` (`0xb2`) to `return` (`0xb1`) — the gate just exits immediately. `dependencies.jar`'s `OS.<clinit>` `startsWith("1.")` branch is also patched (`bipush 8` → `bipush 25`) as belt-and-suspenders in case CheerpJ ever switches to reporting `"1.8.0_xxx"`. Both patched JARs are committed; python only runs when a JAR was just downloaded fresh.
- Mindustry `v157.4` ships both `Mindustry.jar` and `dependencies.jar`; the web launcher now uses an explicit classpath plus `cheerpjRunMain(...)`.
- Arc's `System.loadLibrary("arc")` and `System.loadLibrary("sdl-arc")` calls are now routed through `/app/natives` JS modules via `java.library.path=/app/natives`.
- `override/scripts/global.js` now shadows Mindustry's bundled Rhino bootstrap with a much smaller compatibility script so startup can proceed without full JavaScript mod support.
- The current web port only shims the SDL / OpenGL native layer. Arc native libraries such as FreeType and Soloud are still a larger compatibility gap.
- `arc.util.SharedLibraryLoader.load` is patched to an immediate `return`. Mindustry calls `System.loadLibrary("arc64")` / `sdl-arc64` / `arc-freetype64` / `arc-filedialogs64` through it; CheerpJ then tries to dynamic-`import()` a JS file from `java.library.path` and creates the `Blob` without a MIME type, which Chrome rejects (`Failed to load module script: ... MIME type of "null"`) and which we can't patch from the main thread because the import runs in a CheerpJ worker. Skipping `loadLibrary` entirely works because the actual JNI methods are registered globally via the `natives` option to `cheerpjInit`, so `Java_arc_backend_sdl_jni_SDL_*` resolves directly without ever needing a "library" object.
- If you still see `ERR_CACHE_OPERATION_NOT_SUPPORTED`, test with browser cache disabling turned off and clear any old service workers registered for this site.
